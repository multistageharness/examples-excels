# Skill tests

Two suites over the skills in [`../.agents/skills/`](../.agents/skills/), answering two
different questions:

| Suite | Question | Model? |
| --- | --- | --- |
| `skills-loader` + `skills-harness` | Does the model, handed a skill, **say** the right thing? | yes (mocked or live) |
| `skills-exec` | Does the skill, when **run**, do the right thing? | no — it runs the scripts |

The second is the one that catches broken code. It spawns each skill's real entrypoint against
real workbooks and asserts on the bytes that come back; nothing in it is mocked. See
[What `skills-exec` actually runs](#what-skills-exec-actually-runs).

The SDK is vendored (not installed) at [`../../vendor/github-copilot-sdk`](../../vendor/github-copilot-sdk)
— a copy of `packages/ts` from
[multistageharness/llm-sdk-github-copilot](https://github.com/multistageharness/llm-sdk-github-copilot),
pinned at the commit recorded in that directory's `.vendor-commit`.

```bash
npm test          # mocked — no network, no CLI, nothing to install
npm run test:live # the same assertions against the real Copilot model
```

## Why the mocked run needs nothing installed

`CopilotHarness` imports `@github/copilot-sdk` **lazily**, and only when its
`deps.clientFactory` seam is empty. The tests fill that seam with the SDK's own
`MockClient`, so the default run never spawns the Copilot CLI, never opens a socket, and
never needs `npm install`.

`test:live` removes the seam. That path *does* require `@github/copilot-sdk` on disk and an
authenticated Copilot CLI; without them it fails with `ERR_MODULE_NOT_FOUND`.

## What "confirm output" means

A mocked model that replies with the answer the test scripted would confirm nothing. So the
confirmation is built in layers, and each layer is independently able to fail:

| Layer | Asserted in | Fails when |
| --- | --- | --- |
| The skill is **loadable** and self-consistent | `skills-loader.test.mjs` | frontmatter is malformed, `name` ≠ directory, a dependency does not resolve, the graph has a cycle |
| The golden is **grounded in the skill text** | `skills-loader.test.mjs` | a phrase the expected answer relies on is no longer in that `SKILL.md`, the golden entrypoint does not exist on disk, or the golden contradicts `metadata.role` / declared `dependencies` |
| The skill **reached the model** | `skills-harness.test.mjs` | the session's system prompt does not contain the skill body verbatim, or a dependency skill was not loaded alongside it |
| The reply **satisfies the schema** | `skills-harness.test.mjs` | the harness accepted an off-schema answer, or failed to re-ask on an invalid one |
| The reply **is the golden** | `skills-harness.test.mjs` | the answer is schema-valid but wrong (e.g. a `py-` skill reported as `bash`) |

The goldens live in [`helpers/expectations.mjs`](helpers/expectations.mjs) and drive **both**
modes — the mocked run and the live run assert the identical expected values.

## The grounding guard

The weak point of a golden is that it drifts into fiction: someone edits a skill, and the
expected answer quietly stops being derivable from the skill's own text. Every expectation
therefore carries a `grounding` list of phrases that **must** appear in that skill's body. Rename
`--detect-modified` in a `SKILL.md` and the loader test fails, pointing at the golden that just
became a lie — rather than the harness test continuing to pass against text that no longer says
what it claims.

## Layout

```
tests/
  helpers/
    skills.mjs        # SKILL.md loader: frontmatter parser, dependency graph, prompt renderer
    expectations.mjs  # the goldens + JSON schemas + grounding phrases
    harness.mjs       # binds a skill into the vendored harness (mock seam / live)
    exec.mjs          # spawns the skills' real scripts; fixture generation; the golden rows
  skills-loader.test.mjs   # the tree is loadable and the goldens are honest — no model
  skills-harness.test.mjs  # load each skill through the SDK, confirm the output
  skills-exec.test.mjs     # RUN each skill and confirm the output — no model at all
```

## What `skills-exec` actually runs

Every one of the 11 skills, end to end. Fixtures are generated by the suite's own
`make_fixtures.py`, so the workbooks are produced by the skill under test rather than by a
committed copy that could drift. `sample.xlsx` has a golden diff, quoted from
`py-xlsx-diff-commons/SKILL.md`:

```
Feb  103  Removed   Active    Charlie
Feb  105  Added     Active    Eve
Mar  102  Removed   Inactive  Bob      <- March's row, February's values
Mar  106  Added     New       Frank
```

That third row is the whole point. A `Removed` row is labelled with the month the removal was
**observed** in, but carries the values from the month **before** it — the last month the row
actually existed. Get it backwards and you still get a plausible-looking file.

Two things make these tests load-bearing rather than merely green:

**Negative controls.** Each verifier is fed a deliberately corrupted file as well as an honest
one — Bob's `Removed` row is rewritten to carry January's `Active` instead of February's
`Inactive` — and must reject it *and name the rule*. A verifier that cannot fail proves nothing.

**Cross-implementation checks.** The bash suite and the Python suite share no code, so each
reading back the other's output is the strongest evidence available that either is right. The
suite asserts byte-identical stdout across both engines (4 fixtures × 2 formats), **and equal
exit codes** — plus each engine's verifier accepting the other engine's `.xlsx`.

### Requirements

A Python with `openpyxl`. The suite looks for the reference project's virtualenv, then
`python3`; override with `XLDIFF_PYTHON=/path/to/python`. It **fails rather than skips** when it
cannot find one — a silently-skipped execution suite is indistinguishable from a passing one.
The bash skills need only `unzip`, `zip`, `sed`, `awk`, `sort`, `cut`.

### Verified by mutation

Each layer was proven able to fail by breaking the code under it:

| Mutation | Tests that caught it |
| --- | --- |
| `Removed` row carries blanks instead of the previous month's values | 16 |
| `table`/`csv`/`json` exit 1 on an empty diff (the bug found and fixed here) | 3 |
| `Removed` fill colour changed from red | 1 |
| M injector copies the stale length prefix instead of recomputing it | 1 |

That last one initially caught **nothing** — `inject_m.py`'s own read-back slices the mashup
package *by* the declared length and only checks that the slice starts with `PK`, so an
overshooting prefix silently swallows trailer bytes, still parses, and still prints
`M verified`. The suite now checks the invariant directly: the declared length must exactly
cover the package ZIP, with zero slack. Excel enforces this; nothing else in the repo did.

## Coverage

All 11 skills are probed for `runtime`, `role`, `entrypoint`, and `dependencies`. The two skills
whose text states the full diff contract (`py-xlsx-diff-commons`, `py-xlsx-month-diff`) are
additionally probed on the three rules a wrong-but-plausible diff gets backwards:

- a `Removed` row carries the **previous** month's values, not the current month's
- a gap in the month sequence yields an **empty diff**, not an error
- `Modified` detection is **opt-in** (`--detect-modified`)

Harness-contract behaviors (fenced-JSON extraction, the repair round-trip, hard failure after
exhausted repairs, hallucinated-entrypoint rejection, the token-budget preflight, and the
empty-prompt guard) are pinned in mocked mode only — against a live model they would be
nondeterministic, so they are skipped when `COPILOT_LIVE=1`.
